주제명
이기종 엣지 AI 환경을 위한 Agent-Assisted 동적 오프로딩 및 런타임 오케스트레이션

영문 제목 후보
Heterogeneity-Aware Runtime Orchestration with Agent-Based Planning for Edge AI Systems

1. 왜 이 주제를 하는가
온디바이스 AI와 엣지 AI가 확산되면서, 단일 서버가 모든 연산을 처리하는 구조는 점점 한계를 보이고 있다. 실제 서비스는 이제 디바이스–엣지–클라우드가 함께 협력하는 분산 실행 구조를 필요로 한다.

하지만 실제 엣지 환경은 다음 문제를 가진다.

디바이스마다 CPU, 메모리, GPU/NPU, 네트워크 상태가 다름

서비스 부하가 시간에 따라 계속 변함

정적 배치나 단순 규칙 기반 오프로딩은 병목을 쉽게 유발함

AI 서비스를 통째로 옮기는 coarse-grained offloading은 비효율적임

mixed-device 환경에서는 어떤 stage는 디바이스에, 어떤 stage는 edge 서버에 두는 식의 세밀한 분할이 필요함

즉, 이 연구는 이기종 엣지 환경에서 AI 서비스를 어떻게 동적으로 나누고, 어디에 배치하고, 언제 재배치할 것인가를 다룬다.

2. 핵심 문제정의
한 줄 문제정의
제한된 자원과 서로 다른 하드웨어 구성을 가진 이기종 엣지 환경에서, 시스템 상태를 반영하여 AI 서비스 작업을 동적으로 분할·배치·오프로딩하고, 이를 자율적 planning 기능으로 보조할 때 기존 정적 또는 heuristic 방식보다 더 낮은 지연 시간과 더 안정적인 자원 활용을 달성할 수 있는가?

상세 문제
이 연구가 겨냥하는 문제는 4가지다.

(1) 정적 배치의 한계
처음에 잘 맞는 배치도 시간이 지나면 부하 변화 때문에 금방 비효율적이 된다.

(2) 이기종성으로 인한 불균형
x86 서버, Jetson, Raspberry Pi 같은 노드는 성능과 자원 구성이 달라서 같은 정책을 일괄 적용하면 병목이 생긴다.

(3) 기존 오프로딩 state의 한계
CPU 사용률, 메모리 사용률 같은 단순 지표만으로는 실제 병목과 서비스 지연 원인을 제대로 설명하지 못한다.

(4) planning 기반 제어 부족
기존 스케줄러는 현재 여유가 있는 노드를 고르는 수준에 머무는 경우가 많다. 하지만 실제로는 workflow를 다시 나누거나, 특정 stage만 이동하거나, 복제를 늘리는 식의 재계획이 필요하다.

3. 이 연구의 핵심 아이디어
이 연구의 핵심은 다음이다.

핵심 문장
AI 서비스를 workflow 단위로 분해하고, 이기종 노드 상태를 반영하는 runtime orchestration engine이 기본 배치를 수행하며, planning이 필요한 경우에만 agent-assisted layer가 재배치와 오프로딩 결정을 보조한다.

즉,

메인 기여는 runtime orchestration system

agent는 보조적인 planning/policy layer

로 둔다.

중요한 점은 “Agentic AI가 모든 걸 다 한다”가 아니라는 것이다.
그렇게 잡으면 범위가 너무 커지고 논문이 약해진다.

4. 시스템 구조
시스템은 크게 4계층으로 본다.

4.1 Runtime State Monitor
각 노드와 서비스의 상태를 수집한다.

예시:

CPU usage

memory pressure

accelerator availability

node type

network throughput / latency proxy

queue length 또는 backlog proxy

stage별 실행 시간

end-to-end latency

목적:

의사결정 가능한 상태 벡터 생성

4.2 Workflow Decomposer
AI 서비스를 분할 가능한 workflow DAG로 표현한다.

예시:

capture

preprocess

inference

postprocess

result delivery

목적:

서비스 전체 이동이 아니라 stage-level offloading 가능하게 만들기

4.3 Heterogeneity-Aware Placement Engine
기본 배치와 재배치를 수행하는 핵심 엔진이다.

입력:

노드 capability/profile

stage 요구 자원

현재 runtime state

출력:

각 stage의 배치 위치

migration/replication 여부

offloading 여부

예시 비용함수:
Cost(stage, node) = compute_delay + transfer_cost + memory_pressure + migration_penalty

목적:

기본적인 runtime orchestration 수행

4.4 Agent-Assisted Planning Layer
planning이 필요한 상황에서만 개입하는 보조 계층이다.

개입 조건 예시:

workload burst 발생

p95 latency 급증

특정 노드 overload

반복 병목 발생

SLA violation 가능성 증가

장애/성능 저하 발생

출력 예시:

workflow 재분할 여부

특정 stage의 sibling edge 이동 여부

cloud offloading 여부

replica 증가 여부

degraded mode 전환 여부

목적:

정상 상태는 placement engine이 처리하고,

급격한 변화가 있을 때만 planning 기능으로 재구성 보조

5. 전체 동작 흐름
전체 루프는 아래처럼 정리된다.

monitor → model → place → replan → execute → feedback

단계별 흐름
각 노드의 자원 상태와 capability를 수집

AI 서비스를 workflow stage로 모델링

placement engine이 기본 배치 수행

병목이나 workload 변화가 크면 agent-assisted layer가 재계획

새 배치 실행

결과를 다시 상태로 반영

6. 연구 목표
목표 1
이기종 edge 노드의 상태와 capability를 반영할 수 있는 runtime state model 설계

목표 2
AI 서비스를 workflow 단위로 분해하고 stage-level dynamic offloading이 가능한 orchestration framework 구현

목표 3
기본 placement engine 위에 agent-assisted planning layer를 결합하여 burst workload와 병목 상황에서 재계획 능력 검증

목표 4
정적 배치, heuristic 오프로딩, agent-assisted orchestration을 비교하여 latency, makespan, 자원 활용률, migration overhead 측면의 효과 검증

7. 기대 기여점
기여점 1
이기종 mixed-device edge 환경에서 동작하는 workflow-aware runtime orchestration framework 제안

기여점 2
정적/heuristic 방식보다 더 세밀한 stage-level dynamic offloading 및 재배치 메커니즘 제시

기여점 3
planning이 필요한 상황에서만 개입하는 agent-assisted policy layer를 통해 급격한 상태 변화 대응 능력 향상

기여점 4
실제 mixed-device testbed에서 latency, makespan, load balance, migration overhead 기준으로 실험적 검증

8. 왜 이 주제가 괜찮은가
이 주제가 좋은 이유는 다음과 같다.

(1) 현재 트렌드와 맞음
최근 edge/IoT/AI systems는

heterogeneous orchestration

dynamic offloading

device-edge-cloud continuum

autonomous planning

방향으로 가고 있다.

(2) 당신 과제와 정합성이 높음
과제 문서 자체가 이미

AI 서비스 분할 및 오프로딩

통합 자원 관리

멀티 디바이스 오케스트레이션

대규모 mixed-device 관리

를 핵심 범위로 둔다.

(3) 실험 환경이 있음
이미 x86 + Jetson + Raspberry Pi 기반 KubeEdge 환경이 있으므로 실증 가능성이 높다.

(4) agent를 과장하지 않아 안전함
“Agentic AI”를 메인 기여로 내세우지 않고, planning 보조 계층으로 제한했기 때문에 범위 통제가 가능하다.

9. 주의해야 할 점
이 주제를 쓸 때 피해야 할 표현도 있다.

피해야 할 것
“Agentic AI가 전체 시스템을 자율적으로 운영한다”

“LLM이 스케줄링을 완전히 대체한다”

“최초의 agentic edge orchestrator”

“모든 상태를 완벽히 이해해 최적 배치를 수행한다”

안전한 표현
agent-assisted planning

autonomous replanning

runtime orchestration

workflow-aware placement

heterogeneity-aware offloading

즉, 메인 기여는 시스템, agent는 제한된 planning layer로 유지해야 한다.

10. 최종 한 줄 요약
짧은 요약
이 연구는 mixed-device edge AI 환경에서 AI 서비스를 workflow 단위로 분해하고, 상태 기반 runtime orchestration과 agent-assisted planning을 결합하여 동적 오프로딩과 재배치를 수행하는 시스템을 제안하는 것이다.

더 짧은 기록용 요약
정적 배치가 어려운 이기종 엣지 환경에서, workflow-aware·state-aware·heterogeneity-aware 오케스트레이션을 구현하고, planning이 필요한 상황에서만 agent-assisted layer가 재배치를 보조하는 구조를 연구한다.

11. 기록용 초안 문단
아래 문단은 메모장에 그대로 저장해도 된다.

본 연구는 이기종 엣지 AI 환경에서의 동적 자원 오프로딩 및 런타임 오케스트레이션 문제를 다룬다. mixed-device edge 환경에서는 노드별 연산 성능, 메모리, 가속기 구성, 네트워크 상태가 상이하고 서비스 부하 또한 시간에 따라 동적으로 변화하므로, 정적 배치나 단순 heuristic 기반 오프로딩만으로는 병목과 지연 시간을 안정적으로 제어하기 어렵다. 이에 따라 본 연구는 AI 서비스를 workflow stage 단위로 분해하고, runtime state를 기반으로 기본 배치를 수행하는 heterogeneity-aware orchestration framework를 설계한다. 또한 workload burst, 반복 병목, SLA 위반 가능성 등 재계획이 필요한 상황에서는 agent-assisted planning layer가 개입하여 workflow 재분할, stage migration, cloud offloading, replication 등의 결정을 보조하도록 한다. 최종적으로 정적 배치 및 기존 heuristic 방식과 비교하여 end-to-end latency, makespan, resource utilization, migration overhead 측면에서 성능 향상을 검증하는 것을 목표로 한다.
